## コスト構造の注意点

| 注意点 | 説明 |
|---|---|
| **LLMコストが支配的** | 竹・松ではBedrock Sonnet 3.5のトークン費用が総コストの50〜65%を占める。プロンプト最適化・RAGコンテキスト圧縮が最大のコスト削減ポイント |
| **梅と竹の価格差が大きい** | OpenSearch Serverlessの最低固定費（~$345/月）とECS Fargate常時稼働により、竹は梅の約40倍のコスト |
| **Bedrockクォータ** | Sonnet 3.5はデフォルト20 req/分。竹・松は事前にAWSへのクォータ引き上げ申請が必要 |
| **為替リスク** | 全サービスがドル建てのため、円安局面でコストが増加する |
| **Shield Advanced** | $3,000/月（約45万円/月）は費用対効果を慎重に検討。組織単位の請求のため、他サービスと共有可 |
| **M365環境での認証管理工数** | CognitoはAzure Entra IDとのSAML連携が必要。Cognitoサービス自体は無料だが、初期構築3〜5人日＋月次1〜2h/月の管理工数が発生。Azure（Entra ID流用でゼロ）と比較してコスト増となる点に注意 |

## M365環境でのリスク（Azureとの比較）

> M365・Entra IDを認証基盤として利用しているクライアントがAWSを採用する場合に固有のリスク

### 運用リスク

| リスク | 深刻度 | 内容 |
|---|---|---|
| **SAML証明書切れによる全社ログイン不可** | 🔴 高 | Entra ID側のSAML証明書（通常3年有効）が期限切れになると全ユーザーがログイン不可になる。CognitoとEntra IDの両側で更新作業が必要で、対応漏れが起きやすい |
| **深夜・休日の認証障害対応** | 🔴 高 | 「ログインできない」障害はユーザー影響が大きく緊急対応が必要。原因がEntra ID側かCognito側か切り分けるだけで時間を要し、MicrosoftとAWS両方へのサポート問い合わせが発生する |
| **M365テナント変更の波及** | 🟡 中 | 社内でM365テナント統合・条件付きアクセスポリシー強化・MFAポリシー変更があるたびに、Cognito側の設定を追随して変更する必要がある |
| **担当者の属人化** | 🟡 中 | Cognito + Entra ID SAML連携を両方理解している担当者が必要。退職・異動時に引き継ぎが困難になり、障害対応や設定変更が止まるリスクがある |
| **2プラットフォームの障害対応コスト** | 🟡 中 | AWS障害はAWSサポートへ、Microsoft障害はMicrosoftサポートへと問い合わせ先が分かれる。障害の原因特定・エスカレーションに時間がかかる |

### 開発・構築リスク

| リスク | 深刻度 | 内容 |
|---|---|---|
| **初期構築の複雑性** | 🟡 中 | Cognito User Pool作成→Entra IDにSAMLアプリ登録→CognitoにIdP設定→属性マッピング設定、と工程が多く設定ミスが起きやすい |
| **スキル調達の難しさ** | 🟡 中 | AWS（Cognito）とMicrosoft（Entra ID）の両方に精通したエンジニアは少ない。SES調達時にスキルセットの条件が厳しくなる |
| **認証テストの複雑化** | 🟠 低〜中 | ログインフローのテストにEntra IDのテストユーザーとCognitoの両方が絡むため、CI/CDでの自動テスト構築が難しい |

### コスト面のリスク

| リスク | 内容 |
|---|---|
| **見えない人件費増** | Cognito自体は無料だが、二重管理による月次+2.8〜6.5hの工数増が人件費として発生する（エンジニア単価80万円/月換算で年間+19〜38万円） |
| **障害対応の長期化コスト** | 認証障害は原因特定に時間がかかるため、Azureと比べてインシデント対応コストが高くなりやすい |
